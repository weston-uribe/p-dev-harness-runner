# v0.4.0 — Associations, fingerprint gate, and release-sync runner

**Release type:** GitHub source release (annotated tag + curated release notes) plus public npm package **`p-dev-harness@0.4.0`**.

**Status:** Release-preparation — not tagged, not published on npm, and no GitHub release until operator approval of the release-prep PR and final artifact checksum.

## TL;DR

`p-dev-harness@0.4.0` hardens first-run Linear→repo association routing, pairs `HARNESS_CONFIG_JSON_B64` with `HARNESS_CONFIG_FINGERPRINT` for fail-closed `cloud_config_stale` checks, speeds Step 1 provisioning, and ships an operator CLI (`release:sync-managed-runner`) to sync a managed harness runner and require the configuration canary. The in-app **Update PDev runner** card remains **disabled by default**.

**Primary install command (after npm publish):**

```bash
npx --yes p-dev-harness@0.4.0
```

Requires Node.js 22+. macOS is the validated packaged platform; use `--no-open` elsewhere.

Full guide: [`docs/p-dev.md`](../p-dev.md)

## What ships

- Authoritative Linear associations for issue routing (Settings + Configure Step 2)
- Paired cloud config secret + repository variable fingerprint (`HARNESS_CONFIG_JSON_B64` / `HARNESS_CONFIG_FINGERPRINT`), including `run-merge`
- Configuration canary with safe failure diagnostics (expected/computed fingerprint, decode + association success) and workflow_dispatch 204 run location via `canary_operation_id`
- Operator CLI: `npm run release:sync-managed-runner`
- Faster Step 1 provisioning (bulk authenticated git push / packaged Git object pack)
- Guided Configure UX holds and Step 7 server-authoritative finalization improvements
- Runner status comments / conditional Blocked finalization refinements
- In-app Update PDev runner implementation **hidden unless** `P_DEV_RUNNER_UPGRADE_UI_ENABLED=1`

## What does not ship

- Enabled-by-default in-app runner upgrade UI (deferred; experience TBD)
- Automatic upgrade of every existing managed workspace without operator action
- Changes to `weston-uribe/p-dev-harness-template`
- Provider-agnostic agent support or production SaaS claims
- npm publish / git tag / GitHub release in the release-prep PR (operator-gated after merge)

## Security and mutation boundaries

- Secrets remain in operator workspace, GitHub Actions secrets/vars, and Vercel env vars — never in the published package
- Published tarball excludes credentials, operator state, and local caches
- Release-prep validation must not publish npm, create tags, or create GitHub releases
- External template repository is not mutated for 0.3.1+

## Validation summary

Release-prep validation includes:

- Full test suite + webhook tests
- `npm run build`
- `package:p-dev:prepare` / `pack` / `inspect`
- `npm audit --audit-level=moderate` (document accepted transitive risk if unchanged)
- Minimal packaged smoke: install tarball, launch server, load `/settings/configure`
- Prior FRE-1 end-to-end release gate on the test workspace (not re-run for this prep PR)

Immutable npm/tag evidence is filled in the post-release finalization PR after operator-approved publish.

## Known limitations

- Cursor-only agent provider
- Linear / GitHub / GitHub Actions / Vercel stack
- macOS validated for packaged browser auto-launch
- In-app runner upgrade UX intentionally unfinished / disabled for 0.4
- Not production SaaS or provider-agnostic

## Rollback

- **Unpublished code:** revert merge commit on `main`
- **Published npm:** follow npm deprecation policy only under explicit operator approval; prefer forward-fix patch for defects
- **Never** overwrite tags or republish the same npm version with different bytes
