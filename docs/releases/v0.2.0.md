# V0.2.0 — Release contract

**Tag status:** Tagged — tag `v0.2.0` on `fe37e307930467d66840624c82bce9dc0b4af0f2`. GitHub release: https://github.com/weston-uribe/agentic-product-development-harness/releases/tag/v0.2.0

**Release type:** GitHub **source release** (annotated tag + curated release notes). This is **not** an npm package publication. `package.json` remains `private: true`.

## TL;DR

V0.2.0 is the first source release of the working harness: Linear-triggered Cursor Cloud Agent runs, GitHub PR lifecycle automation, production sync, and the public-repo security baseline.

It is intentionally narrow:

- Cursor only
- Linear only
- GitHub / GitHub Actions only
- Vercel only where preview/webhook support is configured

It is not a provider-agnostic platform and not an npm package.

## Who this is for

V0.2.0 is for a solo operator or small product/engineering team that wants to run a controlled AI-assisted product-development loop from Linear to GitHub PRs.

It assumes the operator is comfortable configuring GitHub Actions secrets, Vercel environment variables, Linear statuses, and target repo branch strategy.

---

## Summary

V0.2.0 is a Cursor-first harness for orchestrating product work through **Linear + GitHub + GitHub Actions**. SDK runners handle planning through merge and production sync; Linear status changes trigger cloud harness runs via a Vercel webhook bridge and GitHub `repository_dispatch`.

## Security model

- **Secrets** live in GitHub Actions secrets, Vercel environment variables, or local untracked `.env` files — never committed.
- **Public repo:** untrusted users cannot push to `main` or trigger secret-bearing automation without write access or a dispatch token.
- **`main` protection:** ruleset requires PR, required status checks (`test`, `Analyze (javascript-typescript)`), no direct push, no force push.
- **Actions allowlist:** only pinned first-party actions permitted.
- **Output redaction:** harness stdout is redacted before logs, summaries, and artifacts.
- **Accepted risks:** documented in [`docs/security.md`](../security.md) (dispatch token model, transitive `undici` advisory, public Actions logs).

## Solo repo automation policy

While Weston is the sole maintainer:

| Setting | Value |
|---------|-------|
| PR required to merge to `main` | Yes |
| Required status checks | `test`, `Analyze (javascript-typescript)` |
| Direct push to `main` | No |
| Required GitHub approvals | **0** |
| CODEOWNER review enforced | **No** (`.github/CODEOWNERS` exists for documentation/future enforcement) |

**Linear/status gates remain required.** Automation runs only when the issue is on an allowlisted status; merge runs only after **Ready to Merge**. This is human-controlled product judgment via Linear — not bypassed by GitHub review settings.

**Re-enable GitHub approvals and CODEOWNER review** when a second engineer joins or when accepting external contributions.

## Setup prerequisites

1. Node.js 22+, npm
2. Linear team with harness workflow statuses configured
3. GitHub repo with Actions secrets: `LINEAR_API_KEY`, `CURSOR_API_KEY`, `HARNESS_GITHUB_TOKEN`
4. Vercel project for webhook bridge (production env vars only — see security doc)
5. `harness.config.json` with target repos and Linear mappings
6. Optional: integration branch/`main` branch setup and production sync dispatch

## Known limitations

See [CHANGELOG.md](../../CHANGELOG.md) Known limitations section for the full list. Key points:

- **Cursor-only** — no Claude Code, Codex, or local VS Code agent support
- **Linear + GitHub only** — not a generic PM or SCM platform
- **Manual evals** — automated eval contract is deferred; scorecards are human-readable rubrics
