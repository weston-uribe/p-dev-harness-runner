# v0.3.1 — Immutable workspace provisioning

**Release type:** GitHub source release (annotated tag + curated release notes) plus public npm package **`p-dev-harness@0.3.1`**.

## TL;DR

`p-dev-harness@0.3.1` binds each published package version to an **immutable embedded workspace snapshot** and provisions fresh private workspaces from that snapshot without calling the public template repository at runtime. Existing valid `p-dev-harness@0.3.0` managed workspaces reconnect without content rewrite.

**Primary install command:**

```bash
npx --yes p-dev-harness@0.3.1
```

Requires Node.js 22+. macOS is the validated packaged platform; use `--no-open` elsewhere.

Full guide: [`docs/p-dev.md`](../p-dev.md)

## What ships

- Embedded `workspace-snapshot/manifest.json` + `workspace-snapshot/files/**` in the npm tarball
- Package-owned GitHub provisioning (`createUserRepository` + git blob/tree/commit APIs)
- Managed marker v3 with `createdFromPackageSnapshot` provenance
- Default blob-upload concurrency **2** with shared rate-limit coordination and automatic retry/pause
- Legacy reconnect for valid `p-dev-harness@0.3.0` managed workspaces
- Conditional release-impact guidance in planner/implementation contracts

## What does not ship

- Automatic upgrade of existing managed workspaces to the 0.3.1 snapshot
- Changes to `weston-uribe/p-dev-harness-template` default branch or tags
- Provider-agnostic agent support or production SaaS claims

## Embedded snapshot behavior

Each `p-dev-harness@0.3.1` install carries a curated workspace snapshot generated from the release commit. Fresh private `OWNER/p-dev-harness` workspaces are created by uploading snapshot blobs to GitHub and committing the snapshot tree plus a managed marker overlay.

- **Package-to-snapshot immutability:** manifest `sourceCommit`, `snapshotContentId`, `snapshotSha256`, and `gitRootTreeSha1` are recorded in the tarball
- **First-time provisioning** can take several minutes; Configure shows upload progress
- **Rate-limit recovery** is automatic — shared pause coordination across concurrent upload workers; no env var required for the default path
- Optional advanced override: `HARNESS_SNAPSHOT_UPLOAD_CONCURRENCY` (integer `1`–`4`)

## Compatibility

| Workspace type | Behavior |
|----------------|----------|
| Fresh install @0.3.1 | Provisions from embedded snapshot |
| Valid 0.3.0 managed workspace | Reconnects without rewrite |
| Existing workspace on newer snapshot | Not auto-upgraded |

`weston-uribe/p-dev-harness-template` remains a **frozen** compatibility artifact for `p-dev-harness@0.3.0` only.

## Release-impact agent-contract changes

Planner and implementation skills include conditional release-impact analysis for published artifacts. Agents must classify impact but do **not** authorize publish, tag, or deploy actions.

## Security and mutation boundaries

- Secrets remain in operator workspace, GitHub Actions secrets, and Vercel env vars — never in the published package
- Published tarball excludes credentials, operator state, and local caches
- Release validation uses uniquely named private repositories; production `weston-uribe/p-dev-harness` is not mutated
- External template repository is not mutated for 0.3.1+

## Validation summary

Release validation includes:

- Full test suite, webhook tests, package prepare/pack/inspect
- Packed npm README and snapshot manifest correctness
- Exact tarball launcher smoke (`/settings/configure` HTTP 200, stop/relaunch)
- Real-account provisioning gate from installed tarball at default concurrency 2
- Registry smoke after npm publication

Immutable npm, tag, and detailed validation evidence are recorded below and in the post-release finalization PR.

## Known limitations

- Cursor-only agent provider
- Linear / GitHub / GitHub Actions / Vercel stack
- macOS validated for packaged browser auto-launch
- No full isolated npm issue lifecycle validated
- No automatic private harness workspace upgrade/sync to newer snapshots
- Not production SaaS or provider-agnostic

## Rollback

- **Unpublished code:** revert merge commit on `main`
- **Published npm:** follow npm deprecation policy only under explicit operator approval; prefer forward-fix `0.3.2` for defects
- **Never** overwrite tags or republish the same npm version with different bytes
